# OAuth State Parameter

## Introduction

OAuth involves multiple redirects between:

```text
Your App

↓

Google

↓

Your App
```

Because redirects happen through the browser, attackers may attempt to trick users into completing unauthorized OAuth requests.

The OAuth State Parameter was created to prevent this.

---

# The Problem

Imagine an attacker creates a malicious OAuth request.

Flow:

```text
Attacker

↓

Creates OAuth Request

↓

Victim Browser

↓

Google Login
```

The victim may unknowingly complete an OAuth flow started by someone else.

This is a form of:

```text
CSRF

(Cross-Site Request Forgery)
```

---

# What Is The State Parameter?

Definition:

> The state parameter is a random value generated by the application and sent with the OAuth request to verify that the callback belongs to the same login flow.

Think:

```text
Unique Request Identifier
```

---

# OAuth Flow Without State

```text
User

↓

Login With Google

↓

Google

↓

Callback

↓

Application
```

Problem:

```text
How Does App Know

This Callback Belongs

To This Login Request?
```

It doesn't.

---

# OAuth Flow With State

Step 1:

Application generates:

```text
xyz123
```

---

Step 2:

Application sends:

```text
Google Login Request

+

state=xyz123
```

---

Step 3:

Google stores and returns:

```text
state=xyz123
```

during callback.

---

Step 4:

Application compares:

```text
Sent State

↓

xyz123

----------------

Returned State

↓

xyz123
```

Match:

```text
YES
```

OAuth continues.

---

# If State Does Not Match

Example:

```text
Sent State

↓

xyz123

----------------

Returned State

↓

abc999
```

Result:

```text
Reject Request
```

The callback is not trusted.

---

# What Problem Does State Solve?

The state parameter protects against:

```text
CSRF

(Cross-Site Request Forgery)
```

It ensures:

```text
The OAuth Response

Belongs To The OAuth Request

Started By This User
```

---

# Mental Model

Think:

```text
Ticket Number
```

When you submit luggage:

```text
Bag

↓

Ticket Number

↓

Claim Later
```

You verify:

```text
Same Ticket

↓

Same Bag
```

OAuth state works similarly.

---

# Why Is State Important?

Without state:

```text
Application

↓

Cannot Verify

Callback Authenticity
```

With state:

```text
Application

↓

Verifies Callback

↓

Continues Securely
```

---

# Key Takeaways

- State is a random value generated before OAuth starts.
- State is sent to the OAuth provider.
- State is returned during callback.
- Applications verify the returned state.
- State helps prevent CSRF attacks.
- State ensures the callback belongs to the original login request.

---

# One-Line Summary

The OAuth state parameter is a random value used to verify that the callback belongs to the original OAuth request and helps protect against CSRF attacks.
