# Statelessness & Cookies - Detailed Revision Notes

# 1. What is Statelessness?

Statelessness means:

> **The server does not automatically remember information from previous requests.**

Every request is treated as completely independent.

Example:

```text
Request 1

GET /profile

↓

Processed

↓

Forgotten
```

Later:

```text
Request 2

GET /orders

↓

Processed

↓

Forgotten
```

The server does NOT automatically know they came from the same user.

---

# 2. Why Was HTTP Designed to be Stateless?

Benefits:

- Simpler server design
- Better scalability
- Easier load balancing
- Independent request processing
- Better reliability

Every request contains all information needed to process it.

---

# 3. The Problem with Statelessness

Suppose a user logs in:

```text
POST /login

↓

Verified

↓

Success
```

Now the user visits:

```text
GET /profile
```

Server asks:

```text
Who are you?
```

Then:

```text
GET /orders

↓

Who are you?
```

Then:

```text
GET /dashboard

↓

Who are you?
```

Since HTTP is stateless, the server does not remember previous requests automatically.

---

# 4. The Core Authentication Problem

Authentication exists because of statelessness.

Main Question:

> How can the server recognize the same user across multiple independent requests?

Solutions include:

- Cookies
- Sessions
- JWT
- OAuth
- OpenID Connect

---

# 5. What is a Cookie?

A Cookie is:

> A small piece of data stored by the browser and automatically sent with future requests to the same server.

Cookies help preserve information across multiple requests.

---

# 6. Cookie Flow

Step 1

```text
Browser

↓

POST /login

↓

Server
```

---

Step 2

Server verifies credentials.

Returns:

```http
Set-Cookie: session=abc123
```

---

Step 3

Browser stores:

```text
session=abc123
```

---

Step 4

Future request:

```http
GET /profile

Cookie: session=abc123
```

Browser sends it automatically.

---

Step 5

Server reads:

```text
session=abc123

↓

Looks up session

↓

Identifies user

↓

Returns profile
```

---

# 7. Who Does What?

Server:

- Creates cookie
- Sends Set-Cookie header

Browser:

- Stores cookie
- Automatically sends Cookie header

Server:

- Reads cookie
- Uses it to identify user

---

# 8. Cookie vs Set-Cookie

Server Response:

```http
Set-Cookie: session=abc123
```

Browser stores it.

Later:

```http
Cookie: session=abc123
```

Browser sends it back.

Memory Trick:

Server → Set-Cookie

Browser → Cookie

---

# 9. Cookies Are NOT Authentication

Cookies themselves do not authenticate users.

Cookies simply:

- Store information
- Automatically send information

Authentication systems often use cookies.

Cookies can store:

- Session IDs
- Language preference
- Theme
- Shopping cart ID
- User preferences

---

# 10. Why Are Cookies Useful?

Without cookies:

```text
Login

↓

Next Request

↓

Login Again

↓

Next Request

↓

Login Again
```

With cookies:

```text
Login Once

↓

Browser Stores Cookie

↓

Automatically Sends Cookie

↓

Server Recognizes User
```

Much smoother user experience.

---

# 11. Browser Automatically Sends Cookies

After storing:

```text
session=abc123
```

The browser automatically includes:

```http
Cookie: session=abc123
```

for matching requests.

The developer usually doesn't manually attach it every time.

# Key Takeaways

- HTTP is stateless.
- Every request is independent.
- Server does not automatically remember users.
- Cookies help preserve information across requests.
- Server creates cookies.
- Browser stores cookies.
- Browser automatically sends cookies.
- Cookies are storage and transmission mechanisms, not authentication.
- Sessions and authentication systems commonly rely on cookies.

---

# One-Line Summary

A cookie is a small piece of data stored by the browser and automatically sent with future requests, helping servers recognize users across stateless HTTP requests.
