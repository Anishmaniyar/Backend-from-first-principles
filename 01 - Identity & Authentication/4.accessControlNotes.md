# Access Control Fundamentals

## What is Access Control?

Access Control is the mechanism that **enforces authorization decisions by allowing or denying access to resources.**

It answers the question:

> **"Should this request actually be allowed?"**

Authentication verifies identity.

Authorization determines permissions.

Access Control enforces those permissions.

---

# Why Do We Need Access Control?

Even if a user is authenticated and their permissions are known, the system still needs a mechanism to enforce those decisions.

Without access control:

```
User

↓

Authenticated

↓

Permission Denied

↓

Still accesses resource ❌
```

With access control:

```
User

↓

Authenticated

↓

Permission Check

↓

Access Control

↓

Allow / Deny
```

Access Control is the actual gatekeeper of the system.

---

# Authentication vs Authorization vs Access Control

## Authentication

Purpose:

```
Who are you?
```

Example:

```
Email

↓

Password

↓

Verified

↓

Authenticated
```

Result:

Identity confirmed.

---

## Authorization

Purpose:

```
What are you allowed to do?
```

Example:

```
Role = Student

↓

Delete User?

↓

No Permission
```

Result:

Permission decision.

---

## Access Control

Purpose:

```
Enforce the permission decision.
```

Example:

```
Permission = Denied

↓

Block Request

↓

403 Forbidden
```

Result:

Request allowed or denied.

---

# Relationship Between Them

```
Identity

↓

Authentication

↓

Identity Verified

↓

Authorization

↓

Permission Decision

↓

Access Control

↓

Allow / Deny

↓

Resource
```

Authentication always comes first.

Authorization depends on authentication.

Access Control enforces authorization.

---

# Why Access Control Exists

Access Control exists to:

- Protect sensitive resources
- Prevent unauthorized actions
- Enforce security policies
- Restrict operations based on permissions
- Prevent privilege escalation

Without access control, security decisions would never be enforced.

# Access Control Using Middleware

Many backend frameworks implement access control through middleware.

Example:

```
Incoming Request

↓

Authentication Middleware

↓

Authorization Middleware

↓

Access Control

↓

Route Handler
```

If checks fail:

```
403 Forbidden
```

If checks pass:

```
Continue Request
```

# Access Control Models (High-Level)

## Role-Based Access Control (RBAC)

Access decisions are based on user roles.

Example:

```
Admin

↓

Delete User

✓
```

```
Student

↓

Delete User

✗
```

---

## Permission-Based Access Control

Access decisions are based on specific permissions.

Example:

```
user:create

user:update

user:delete

post:view
```

---

## Ownership-Based Access Control

Users can only access resources they own.

Example:

```
User A

↓

Own Profile

↓

Can Edit

----------------

User A

↓

User B Profile

↓

Denied
```

# Common Misconceptions

## ❌ Authentication = Access Control

False.

Authentication only verifies identity.

---

## ❌ Authorization = Access Control

False.

Authorization decides permissions.

Access Control enforces them.

---

## ❌ Logged In = Full Access

False.

Authenticated users can still be denied access.

---

## ❌ Admin is Access Control

False.

Admin is simply a role used by an access control system.

---

## ❌ Access Control only uses Roles

False.

Access Control can use:

- Roles
- Permissions
- Ownership
- Policies
- Attributes
- ACLs
- RBAC
- ABAC

---

# Key Takeaways

- Access Control enforces security decisions.
- Authentication verifies identity.
- Authorization determines permissions.
- Access Control allows or denies requests.
- Authentication always happens before authorization.
- Authorization happens before access control enforcement.
- Backend middleware commonly implements access control.
- RBAC is one implementation of access control.
- Authenticated users can still be denied access.
