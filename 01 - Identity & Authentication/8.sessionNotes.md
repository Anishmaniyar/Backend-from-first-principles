# Session Fundamentals

## What is a Session?

A session is a mechanism that allows a server to remember information about a client across multiple HTTP requests.

Sessions solve the problem of HTTP being stateless.

The most common use case is remembering authenticated users after login.

---

# Why Do Sessions Exist?

HTTP is stateless.

Each request is independent.

Example:

```
Request 1

↓

Server

↓

Response

------------------

Request 2

↓

Server

↓

Response
```

The server does not automatically remember:

- Who the user is
- Whether they are logged in
- Previous requests
- Previous interactions

Sessions provide this missing memory.

---

# Stateless HTTP Recap

HTTP treats every request as completely independent.

Example:

```
GET /

↓

Response

-------------------

GET /profile

↓

Response
```

The second request contains no built-in information about the first request.

Without sessions, the server would need to authenticate the user again for every request.

---

# What Problem Do Sessions Solve?

Without sessions:

```
Every Request

↓

Email

↓

Password

↓

Verify Again

↓

Authenticated
```

This would be inefficient and inconvenient.

Sessions allow users to log in once and remain authenticated across future requests.

---

# What is a Session ID?

A Session ID is a unique random identifier generated by the server.

Example:

```
A8f92LmX7Pq81
```

It identifies a particular session but does not itself contain meaningful user information.

The Session ID acts as a reference to session data stored on the server.

---

# High-Level Session Storage

Browser:

```
Session ID
```

Server:

```
Session ID

↓

User Information

↓

Authentication State

↓

Permissions

↓

Expiration Time

↓

Additional Session Data
```

The browser typically stores only the Session ID.

The server stores the actual session information.

# Session Usage

Future request:

```
Browser

↓

Session ID

↓

Server

↓

Lookup Session

↓

Session Exists?

↓

Yes

↓

Authenticated User
```

No password verification is required again.

---

# Session Expiration

Sessions usually have an expiration time.

```
Session Created

↓

Active

↓

Timeout

↓

Session Deleted

↓

User Must Login Again
```

Expiration may occur because of:

- Inactivity
- Absolute timeout
- Manual logout
- Server restart
- Session invalidation

---

# Logout Flow

```
↓

↓

```

# Common Misconceptions

## ❌ Session = Cookie

False.

A session is server-side state.

A cookie is a browser storage mechanism.

Cookies often store Session IDs.

---

## ❌ Session stores passwords

False.

Passwords should never be stored in sessions.

---

## ❌ Session ID contains user information

False.

It is usually just a random identifier.

---

## ❌ HTTP automatically remembers users

False.

HTTP is stateless.

Sessions provide memory.

---

## ❌ Sessions never expire

False.

Sessions can expire, timeout, or be invalidated.

---

# Key Takeaways

- Sessions solve HTTP statelessness.
- Sessions allow users to remain logged in.
- The server generates a unique Session ID.
- The browser typically stores only the Session ID.
- The server stores actual session data.
- Future requests are identified using the Session ID.
- Sessions have a lifecycle: creation, usage, expiration, and deletion.
- Logout invalidates the session.
- Sessions are not the same as cookies.

# One-Line Summary

A session is a server-side mechanism that allows a stateless HTTP protocol to remember users across multiple requests by associating a random Session ID with stored session data.
