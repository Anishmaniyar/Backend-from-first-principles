# Salting Fundamentals

## What is a Salt?

A salt is a **random value generated for each password and added to the password before hashing.**

It ensures that even if two users choose the same password, their stored password hashes will be different.

Simple idea:

```
Password

+

Random Salt

↓

Hash

↓

Store Hash
```

A salt is **not a password** and **not a secret key**.

---

# Why Do We Need Salt?

Hashing alone is not enough.

Without salt:

```
hello123

↓

Hash

↓

ABC123
```

Another user:

```
hello123

↓

Hash

↓

ABC123
```

Same password.

Same hash.

Attackers can immediately identify users with identical passwords.

Salt prevents this.

# What Problems Does Salt Solve?

Salt provides several important security benefits.

## 1. Prevents Identical Passwords From Producing Identical Hashes

Without salt:

```
password123

↓

ABC
```

Everyone:

```
ABC
```

With salt:

```
password123

+

Salt A

↓

AAA
```

```
password123

+

Salt B

↓

BBB
```

Completely different hashes.

---

## 2. Protects Against Rainbow Table Attacks

Attackers often precompute:

```
password

↓

Hash
```

for millions of common passwords.

Example:

| Password | Hash   |
| -------- | ------ |
| 123456   | AAA111 |
| password | BBB222 |
| hello123 | CCC333 |

When a database leaks:

```
Stored Hash

↓

Lookup

↓

Password
```

With unique salts, these precomputed tables become impractical.

---

## 3. Prevents Detection of Shared Passwords

Without salt:

```
User A

↓

ABC123
```

```
User B

↓

ABC123
```

Attacker immediately knows:

```
Same Password
```

With salt:

```
User A

↓

XYZ111
```

```
User B

↓

LMN999
```

No relationship is visible.

---

# What is a Rainbow Table?

A rainbow table is a precomputed lookup table that maps passwords to their hash values.

Example:

```
password

↓

Hash

↓

BBB222
```

Instead of computing hashes repeatedly, attackers simply search the table.

```
Hash

↓

Lookup

↓

Original Password
```

Salt makes these tables practically useless.

---

# Where Is Salt Stored?

Salt is normally stored alongside the password hash.

Example:

| Email           | Salt  | Password Hash |
| --------------- | ----- | ------------- |
| anish@gmail.com | X7K91 | 98ABC...      |

This is completely normal.

Salt is NOT required to be secret.

# Common Misconceptions

## ❌ Salt is secret

False.

Salt is usually stored in the database.

---

## ❌ Salt encrypts passwords

False.

Salt is simply random data added before hashing.

---

## ❌ Same password should always produce same stored hash

False.

Different salts produce different hashes.

---

## ❌ Every user should share one salt

False.

Each password should have its own unique random salt.

---

## ❌ Salt alone prevents all attacks

False.

Salt protects against precomputed attacks and identical hashes.

Modern password storage also requires:

- bcrypt
- Argon2
- Slow hashing
- Good password policies

---

# Key Takeaways

- Salt is random data added before hashing.
- Every password should have a unique salt.
- Salt prevents identical passwords from producing identical hashes.
- Salt protects against rainbow table attacks.
- Salt is usually stored in the database.
- Salt does not need to be secret.
- Salt works together with secure password hashing algorithms like bcrypt and Argon2.
