# Trust & Security Mindset

## Introduction

One of the most important principles in backend development is:

> **Never trust anything that comes from the client.**

The frontend exists for user experience.

The backend exists for:

- Authentication
- Authorization
- Validation
- Business logic
- Security
- Data integrity

Every piece of client-provided data should be treated as potentially malicious until verified.

---

# Why Can't Servers Trust Browsers?

The browser is controlled by the user, not by your backend.

Users can modify:

- HTML
- CSS
- JavaScript
- Network requests
- Cookies
- Local Storage
- Session Storage
- Request body
- Query parameters
- URL parameters
- Headers

The browser is not a trusted environment.

Think of every incoming request as coming from an unknown machine on the internet.

```
Unknown Client

↓

Your Backend
```

Never assume the request was generated by your frontend.

---

# Users Can Completely Bypass Your UI

Your website may provide:

```
Buy Product

Price: ₹1000
```

But an attacker can send:

```json
{
  "price": 1
}
```

using:

- Postman
- curl
- Python scripts
- Their own frontend
- Browser DevTools
- Modified JavaScript

The backend must never trust prices sent by clients.

---

# Real Example: Product Price

❌ Wrong

```json
{
  "productId": 5,
  "price": 1
}
```

Backend:

```
Take Client Price

↓

Charge ₹1
```

Very insecure.

---

✅ Correct

Client:

```json
{
  "productId": 5
}
```

Backend:

```
Read Product ID

↓

Fetch Price From Database

↓

Charge Correct Amount
```

The backend should calculate trusted values itself.

---

# Real Example: User Role

Client sends:

```json
{
  "role": "admin"
}
```

Never trust this.

Correct approach:

```
Session / JWT

↓

User ID

↓

Database

↓

Actual Role

↓

Authorization Check
```

Roles should always come from trusted backend data.

---

# Real Example: Student Marks

Client sends:

```json
{
  "marks": 1000
}
```

Backend should never blindly accept it.

Business rules must always be enforced server-side.

# Golden Rule

Never trust:

- Prices
- Roles
- Permissions
- User IDs
- Account balances
- Discounts
- Request body
- Query parameters
- URL parameters
- Headers
- Cookies
- Local Storage
- Hidden form fields

Everything can be modified.

---

# Why Can't Browsers Trust Servers?

Trust problems exist in both directions.

Not every server on the internet is trustworthy.

A malicious server could attempt to:

- Steal data
- Execute malicious scripts
- Trick users
- Deliver malware
- Perform phishing attacks

Browsers include security mechanisms to protect users.

---

# Browser Security Features

Modern browsers implement protections such as:

- HTTPS verification
- TLS certificate validation
- Same-Origin Policy
- CORS
- Content Security Policy (CSP)
- Sandboxing
- Permission prompts
- Secure cookie handling

These mechanisms reduce the risk of malicious servers harming users.

---

# Never Trust Client Input

Every client-provided value should be considered untrusted.

Examples:

```
Request Body

Query Parameters

Headers

Cookies

URL Parameters

JSON Data

Form Data

Multipart Data
```

Everything must be validated.

---

# Example

Client sends:

```json
{
  "discount": 100
}
```

Backend should verify:

- Is discount allowed?
- Is user eligible?
- Does coupon exist?
- Has coupon expired?
- Has coupon already been used?

Never trust the client.

---

# Client-Side Validation

Client-side validation happens inside the browser.

Example:

```javascript
if (password.length < 8) {
  showError();
}
```

Purpose:

- Better user experience
- Faster feedback
- Fewer unnecessary requests

Client-side validation is NOT a security mechanism.

Users can disable or bypass it.

---

# Server-Side Validation

Server-side validation happens on the backend.

Example:

```
Incoming Request

↓

Validate Data

↓

Reject Invalid Input

↓

Continue Processing
```

Server-side validation provides actual security.

It cannot be skipped by users.

---

# Client Validation vs Server Validation

## Client-Side Validation

Purpose:

- User experience
- Immediate feedback
- Convenience

Examples:

- Required fields
- Password length
- Email format
- Live validation

Can be bypassed.

Never trust it.

---

## Server-Side Validation

Purpose:

- Security
- Business rules
- Data integrity

Examples:

- Email uniqueness
- Password policy
- User permissions
- Age restrictions
- Payment verification
- Database constraints

Must always exist.

---

# Example

Frontend:

```
Password Length >= 8
```

Attacker bypasses JavaScript:

```
"a"
```

Server validation:

```
Length >= 8 ?

↓

Reject
```

Without server validation:

Invalid data enters the database.

# Backend Security Thinking

For every request, ask:

```
What if the client is lying?
```

Examples:

```
What if user changed price?

What if role is fake?

What if userId belongs to another user?

What if amount is negative?

What if coupon is invalid?

What if request was modified?
```

The backend should independently verify everything important.

---

# Security Mindset

Always assume:

```
Client Input

↓

Potentially Malicious

↓

Validate

↓

Authenticate

↓

Authorize

↓

Verify

↓

```

Security is about verifying assumptions instead of trusting input.

---

# Common Misconceptions

## ❌ My frontend generated this request, so it's safe.

False.

Requests can be modified or created manually.

---

## ❌ Hidden HTML fields are secure.

False.

Users can modify hidden fields.

---

## ❌ JavaScript validation guarantees security.

False.

JavaScript can be bypassed or modified.

---

## ❌ Users cannot change request bodies.

False.

Tools like Postman and browser DevTools make this trivial.

---

## ❌ The browser is trusted.

False.

The browser belongs to the user.

---

## ❌ The backend can trust prices sent by the client.

False.

Prices should come from trusted backend data.

---

# Key Takeaways

- Never trust client input.
- The browser is controlled by the user.
- Every request may be malicious.
- Always validate on the server.
- Client-side validation improves UX, not security.
- Business rules belong on the backend.
- Authentication and authorization must be enforced server-side.
- Trusted values should come from the backend or database, not from the client.
- Security begins with verification, not trust.
