# Authorization Fundamentals

## What is Authorization?

Authorization is the process of determining **what an authenticated user is allowed to do or access.**

It answers the question:

> **"What are you allowed to do?"**

Authentication verifies identity.

Authorization verifies permissions.

---

# Why Do We Need Authorization?

Authentication only proves that a user is genuine.

It does **not** determine what actions they can perform.

Example:

```
User logs in

↓

Authenticated

↓

Can they delete all users?

↓

Authorization decides
```

Without authorization:

- Every user could delete data.
- Every user could access admin features.
- Every user could view private information.
- Every user would have unlimited access.

Authorization protects resources by enforcing permissions.

---

# Authentication vs Authorization

## Authentication

Answers:

```
Who are you?
```

Example:

```
Email

↓

Password

↓

Verified

↓

Authenticated
```

Purpose:

Verify identity.

---

## Authorization

Answers:

```
What are you allowed to do?
```

Example:

```
Authenticated

↓

Role = Student

↓

Permission Check

↓

Delete Database?

↓

Denied
```

Purpose:

Determine permissions.

---

# Authentication → Authorization Flow

```
User

↓

Claims Identity

↓

Authentication

↓

Identity Verified

↓

Authorization

↓

Permission Check

↓

Allow / Deny

↓

Access Resource
```

Authorization always happens **after** authentication.

---

# What are Permissions?

A permission is:

> **The right to perform a specific action on a specific resource.**

Examples:

```
Read Profile

Edit Profile

Delete Profile

Create User

Delete User

View Dashboard

Create Post

Delete Post
```

Permissions define exactly what actions are allowed.

---

# Common Permission Naming

A common convention is:

```
resource:action
```

Examples:

```
user:create

user:update

user:delete

user:view

post:create

post:update

post:delete

post:view
```

This naming style is widely used in real-world backend systems.

---

# What are Roles?

A role is:

> **A collection of permissions assigned to a group of users.**

Instead of assigning hundreds of permissions individually, users are assigned roles.

Examples:

```
Student

Teacher

Admin

Moderator

Editor
```

Each role contains a predefined set of permissions.

# Common Misconceptions

## ❌ Authentication and Authorization are the same

No.

Authentication verifies identity.

Authorization verifies permissions.

---

## ❌ Logged in users can access everything

False.

Being authenticated does not automatically grant permissions.

---

## ❌ Admin is a permission

False.

Admin is a role.

Permissions are actions like:

```
user:create

user:update

user:delete
```

---

## ❌ Roles and Permissions are the same

False.

Role:

```
Admin
```

Permission:

```
user:create
```

---

## ❌ Authorization happens before authentication

Impossible.

The system must first know **who you are** before deciding **what you can do**.

Correct order:

```
Identity

↓

Authentication

↓

Authorization

↓

Access Control
```

---

# Key Takeaways

- Authentication verifies identity.
- Authorization determines permissions.
- Authorization always happens after authentication.
- Permissions define specific actions.
- Roles group related permissions.
- Users are typically assigned roles, not individual permissions.
- Backend systems check permissions before allowing actions.
- Unauthorized actions are typically rejected with `403 Forbidden`.

---

# One-Line Summary

Authorization is the process of determining what an authenticated user is allowed to do by checking their roles and permissions before granting access to resources or actions.
