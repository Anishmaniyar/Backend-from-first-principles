# Core Learning

## 1. HTTP is Stateless

Every HTTP request is independent.

The server does not automatically remember previous requests or whether a user has already logged in.

```
Request 1

↓

Server

↓

Response

----------------

Request 2

↓

Server

↓

Response
```

Sessions solve this problem.

---

# 2. What is a Session?

A session is server-side state used to remember an authenticated user across multiple requests.

The session itself lives on the backend.

---

# 3. Session ID

A Session ID is:

- Random
- Unique
- Unpredictable
- Generated by the server

It acts as a lookup key for session data.

Example:

```
abc123xyz

↓

{
    userId: 1
}
```

The Session ID itself contains no meaningful user information.

---

# 4. Session Store

The Session Store maps:

```
Session ID

↓

Session Data
```

Possible storage mechanisms:

- Memory
- Redis
- Database

For this project, an in-memory object was used.

Example:

```
sessions = {

    "abc123": {
        userId: 1
    },

    "xyz456": {
        userId: 2
    }

}
```

---

# 5. Cookies

Cookies are small browser-managed key-value storage.

They are automatically sent with future requests.

Example:

```
session=abc123xyz

theme=dark

language=en
```

Cookies are not sessions.

In session-based authentication, they usually store only the Session ID.

---

# 6. Set-Cookie

The server sends a `Set-Cookie` response header to instruct the browser to store a cookie.

Flow:

```
Server

↓

Set-Cookie

↓

Browser Stores Cookie

↓

Future Requests

↓

Browser Automatically Sends Cookie
```

---

# 7. Cookie Security

## HttpOnly

- Prevents JavaScript from reading cookies
- Helps reduce XSS-based cookie theft

---

## Secure

- Cookie is only sent over HTTPS
- Prevents transmission over insecure HTTP

---

## SameSite

Controls when cookies are sent during cross-site requests.

Modes:

- Strict
- Lax
- None

Helps reduce CSRF attacks.

# 9. Mini Session System

Implemented a custom in-memory session system without using `express-session`.

Authentication Flow:

```
Login

↓

Verify Password

↓

Generate Session ID

↓

Store Session

↓

Set-Cookie

↓

Browser Stores Cookie

↓

Future Request

↓

Browser Sends Cookie

↓

Read Session ID

↓

Lookup Session

↓

Authenticated
```

Logout Flow:

```
Logout

↓

Read Session ID

↓

Delete Session

↓

Remove Cookie

↓

User Logged Out
```

---

# Key Takeaways

- Sessions solve the stateless nature of HTTP.
- The server owns the session.
- The browser typically stores only the Session ID.
- Cookies are not authentication mechanisms.
- `Set-Cookie` tells the browser to store cookies.
- `HttpOnly`, `Secure`, and `SameSite` improve cookie security.
- Never trust client-controlled data.
- Session IDs should always be random, unique, and unpredictable.
- Understanding the underlying flow is more valuable than relying on authentication libraries.
